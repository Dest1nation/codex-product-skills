# Requirement Interview

Ask one focused question at a time when the requirement is fuzzy. The goal is to help the user think, not to complete a form.

## Thinking Order

1. Why now: business background and trigger.
2. Who: target users, operators, approvers, affected roles.
3. When: core scenario and frequency.
4. What outcome: measurable or observable business result.
5. What boundary: must-have, nice-to-have, excluded.
6. What object: records, orders, tasks, documents, content, accounts, or assets.
7. What dependency: systems, data, approvals, policies, manual operations.
8. What risk: compliance, migration, data quality, adoption, deadline.

## Mode Question

If advancement mode is unclear, ask:

```text
这次我们先按哪种方式推进：MVP 收敛、完整蓝图，还是开放探索？
```

Infer mode when the user's wording is obvious, then state the assumption briefly.

## Coaching Rules

- Prefer "what would change your decision?" over "please provide X".
- If the user gives a solution first, ask what problem or scenario makes it necessary.
- If the user gives a broad goal, ask which user/scenario should be solved first.
- If the user gives many functions, ask which one is the first proof that the requirement works.
